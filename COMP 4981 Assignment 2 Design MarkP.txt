What it does
Goal
Accept command-line input for network configuration on the server’s side, establish IPv4 TCP socket connection between a client and the server, and execute remote system commands. The server also supports I/O redirection and multiple concurrent clients. The client and server handle invalid/unexpected input gracefully, and the server further validates and sanitizes input to prevent code injection or undefined behaviour.
Success condition (observable outcome)
The client sends a command to the server, the server executes the command (such as changing directories or listing files) and returns the standard output or error messages to the client terminal. The server remains active and ready for the next command from the client until the exit command is received.
Invariant (one checkable rule)
The server must always check for built-in commands (cd and exit for this application) before attempting to fork(). This ensures that the main process handles state changes and the child process handles external execution.
How I will test it
Built-in Commands
Built-in Execution (cd)
The program is invoked with the cd command to change the working directory on the server.


Steps:
1. Connect the client to the server
2. Run pwd to check the current folder
3. Run cd /tmp
4. Run pwd again


Expected result:
* The second pwd shows /tmp. The server uses chdir()directly without forking.


What it proves:
* The server’s parser correctly identifies the cd built-in and applies changes to the server’s own environment.
Built-in Execution (exit)
The user signals they want to close the connection.


Steps:
1. Connect the client to the server
2. Run exit into the client

Expected result:
   * The client closes, and the server child process for that connection terminates safely.


What it proves:
   *  The exit built-in is correctly identified and the server performs the necessary cleanup of sockets.
External Binary Execution 
External Command (No arguments)
The program is used to run a standard Linux utility that doesn’t need any flags


Steps:
   1. Connect the client to the server
   2. Run whoami from the client
 
Expected result:
   * The server forks a child process, executes the binary, and returns the username to the client.


What it proves:
   * The process creation with fork and binary replacement with either exec or execve is working.


External Command (With Arguments)
The program is used to run a utility that requires flags and paths.


Steps:
   1. Connect the client to the server
   2. Run ls -la /tmp
Expected Result: 
   * The client displays a formatted list of all files in the /tmp directory


What it proves:
   * The parser correctly splits the command string into an argument array.
I/O Redirection 
Input Redirection (<) 
The server pulls input for a command from a file instead of the client socket.


Steps:
   1. Create a file called input.txt on the server containing “COMP4981”
   2. Run cat < input.txt from the client


Expected result:
   * The client displays “COMP4981” 


What it proves:
   * The server identifies < and uses dup2() to replace the child process’ standard input with the file descriptor
Output Redirection (>) 
The program is used to redirect command output into a file on the server


Steps:
   1. Connect the client to the server
   2. Run echo “Test Data” > debug.txt.
   3. Run cat debug.txt

Expected result:
      * The first command produces no output on the client; the second command displays "Test Data."


What it proves:
      * The server correctly identifies the > operator and uses dup2()to redirect the file descriptor before calling execv() or execve().
Error Redirection (2>) 
The server captures error messages and saves them to a file.


Steps:
      * Run ls /nonexistent_folder 2> error.log from the client
      * Run cat error.log


Expected result:
      * The first command produces no output. The second command displays "No such file or directory".


What it proves:
      * The server identifies 2> and uses dup2() to replace the child process’ standard error (STDERR) with the file descriptor.
Concurrency (Multiple Clients)
Two clients connect to the server at the same time to ensure handling of a reasonable number of sessions.


Steps:
      1. Open two clients and connect them both to the server
      2. Run sleep 10 on Client A
      3. Immediately run hostname on Client B


Expected result:
      * Client B receives the hostname immediately while Client A is still waiting.


What it proves:
      * The server correctly identifies the > operator and uses dup2()to redirect the file descriptor before calling execv() or execve().


Error Handling
Standard Error Output
The client receives error messages directly when a command fails.


Steps:
      1. Connect the client to the server
      2. Run ls /root


Expected result:
      * The client displays "Permission denied" directly in the terminal.


What to proves:
      * The child process correctly pipes standard error back to the client socket by default.


Network Error: Port In Use
The server is started on a port that is already occupied


Steps:
      1. Start a server instance on port 8080
      2. Attempt to start a second server instance on port 8080


Expected result:
      * The second server prints “Address already in use” and exits


What it proves:
      * The server correctly checks the return value of bind() and handles errors.


Network Error: Connection Refused
The client attempts to connect to a server that is not running.


Steps:
      1. Run the client with an IP/Port where no server is listening


Expected result:
      * The client displays “connection refused” and exits.


What it proves:
      * The client correctly checks the return value of connect() and handles errors.


Help Option
The program is invoked with a help option.


Steps:
      1. Run the program with the help flag
      2. Observe standard output
      3. Observe program termination

Expected result:
         * Usage information is displayed, and the program exits without attempting to process or display messages.


What it proves:
         * Explicit user requests for help bypass normal execution and terminate cleanly.


Server FSM
Server State meanings
State
	Meaning
	START
	The program has begun execution, but no work has been performed
	PARSE_ARGS
	Command-line arguments (port number) are being read and validated
	USAGE
	An error in arguments was detected, and usage instructions are shown
	SETUP
	The server socket is created, bound to a port, and set to listen
	ACCEPT
	Parent Loop: The server is blocked, waiting for a new client connection
	READ
	Child Loop: The session is blocked, waiting for command input from the client
	PARSE
	The raw input string is tokenized into command and arguments
	BUILTIN
	An internal command (like cd or exit) is being executed
	FORK_CMD
	The session process is branching to create a sub-process for an external binary
	EXEC_BINARY
	Sub-Child: An external program (e.g., ls) replaces the process image
	WAIT_CMD
	Session Child: The session is paused until the external binary finishes
	CLEANUP_SESSION
	The client socket is closed, and the session prepares to end
	CLEANUP_SERVER
	The master socket is closed, and the server prepares to shutdown
	EXIT_CHILD
	The specific client session process has terminated
	EXIT_SERVER
	The entire server program has terminated
